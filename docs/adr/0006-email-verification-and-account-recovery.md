# ADR-0006：邮箱验证与账户恢复边界

- 状态：Accepted
- 日期：2026-07-20
- 决策人：Logion project owner
- 工作包：`L1-002C`

## 背景

现有注册接口会立即保存密码、创建在线会话和个人 Workspace，并以重复邮箱 `409`
区分已有账户。`users.email_verified_at` 已存在但未参与认证，系统也没有邮件找回、
安全通知或可靠的邮件投递边界。这些行为在受限开发阶段可接受，但不能作为公开注册
的生产身份系统。

直接在现有流程后补一个“验证邮箱”按钮仍有账户预劫持风险：攻击者可以先用受害者
邮箱设置自己知道的密码，受害者随后验证邮箱反而会激活攻击者控制的凭据。因此邮箱
所有权证明必须发生在初始密码生效之前。

## 决策

### 1. 注册与邮箱验证

新增枚举安全的注册流程，注册请求只接收邮箱，不保存或替换密码。无论邮箱是新的、
待验证的、已验证的，服务器都返回相同的 `202` 消息、状态码和可缓存策略；内部是否
创建验证流程由服务端决定。

邮箱确认使用一次性高熵 bearer token。确认请求通过可信 Origin 的 `POST` JSON body
提交 token 和初始密码；服务器在同一事务中消费 token、设置密码、写入
`email_verified_at` 并创建个人 Workspace。确认成功后仍要求显式登录，不把验证链接
当作登录链接，也不由邮件扫描器可触发的 `GET` 改变账户状态。

邮件链接将 token 放在 URL fragment 中；fragment 不发送给 Web 服务器。Web 页面读取后
立即从地址栏移除并通过 JSON body 提交。反向代理、分析、Referer、错误响应和审计禁止
记录 token。

已有 `/api/v1/auth/register` 保留为迁移期兼容接口，不直接删除已发布 OpenAPI。新的 Web
客户端只使用枚举安全流程。生产配置禁止旧接口创建未验证会话；旧接口在禁用后返回固定
迁移错误且不查询邮箱。移除旧契约必须等待客户端兼容窗口并使用新 API 版本。

### 2. 未验证账户与迁移

邮箱验证和账户状态是两个独立维度；`active` 不代表邮箱已验证。未验证账户不能普通登录、
注册 Passkey/TOTP、接受 Workspace 邀请或创建业务数据。登录对未验证、不存在、暂停和错误
密码使用同一外部错误。

迁移不得把历史账户批量标记为已验证。上线采用 expand/enforce 两步：先部署投递、确认端点
和提示；确认投递可靠后再启用生产强制。启用强制后，历史未验证账户的现有 Access Session
可自然过期，但 Refresh、重新登录和邀请接受要求完成验证。不得通过延长旧会话绕过验证。

### 3. 密码找回与 MFA

密码找回请求对所有邮箱返回相同 `202`，只为已验证且可登录的账户创建流程。完成请求必须
提供有效的一次性邮件 token 和新密码。若账户启用了 TOTP，完成密码找回还必须提供未重放
的 TOTP 或恢复码；邮件所有权不能单独降级第二因素。已有 Passkey 会话可走“最近认证后修改
密码”流程，而不是邮件找回。

成功重置密码后撤销该用户的全部在线 Session 与 Refresh Token，保留设备登记但不保留其
在线授权；旧密码和重置 token 立即失效。TOTP、Passkey 和未使用恢复码不自动删除。用户同时
丢失邮箱、Passkey、TOTP 与恢复码时不提供客服绕过；未来人工恢复必须另立 ADR、双人批准并
全程审计。

### 4. Token、限流与并发

- 验证和找回 token 至少使用 256 位随机熵，数据库只保存域分离 HMAC；
- 邮箱验证默认 24 小时、密码找回默认 30 分钟，均由有上下界的配置控制；
- 每个用户和用途只保留一个有效流程，新请求撤销旧流程；消费时锁行并原子写 `used_at`；
- 请求端同时使用 IP 桶和 HMAC 邮箱桶，确认端使用 IP 桶和 token 流程失败计数；
- 过期、撤销、已使用、错误用途和未知 token 返回同一错误，不泄漏用户或用途；
- 并发确认最多一个成功，事务失败不得出现“token 已消费但密码/Workspace 未创建”。

### 5. 可靠邮件投递

身份用例只写事务性邮件 Outbox，不直接依赖具体邮件 SDK。Outbox ID 是 Provider 幂等键；
Worker 通过 `EmailDeliveryPort` 投递，记录租约、尝试次数、发送状态和最小化 Provider 回执。

异步投递需要短暂持有 token 和收件地址，因此 Outbox payload 必须使用独立、版本化的投递
密钥进行认证加密，AAD 绑定 outbox ID、用户、用途和密钥版本。不得复用 TOTP KEK。成功发送、
永久失败或过期后清除密文；日志、指标、审计、死信和 Provider 元数据均不得包含 token 或完整
邮箱。开发环境使用可检查的本地投递适配器，生产 Provider 选择不属于本 ADR。

### 6. 安全通知与审计

邮箱确认、密码重置、密码变更和恢复失败上限触发安全通知。安全动作的数据库提交不依赖通知
实时成功；失败进入可重试 Outbox。审计只记录 actor/user ID、流程/Outbox ID、用途、结果和
粗粒度原因，不记录 token、密码、验证码、完整邮箱或邮件正文。

## API 迁移方向

后续实现采用新增而非直接破坏旧端点：

- `POST /api/v1/auth/registrations`：统一 `202`；
- `POST /api/v1/auth/email-verification/confirmations`：消费 token 并设置初始密码；
- `POST /api/v1/auth/password-recovery/requests`：统一 `202`；
- `POST /api/v1/auth/password-recovery/completions`：token、新密码及可选 MFA 证明；
- 已认证的密码变更、安全通知状态和重发策略拆为后续小型工作包。

所有敏感响应使用 `Cache-Control: no-store`。状态改变端点继续要求可信 Origin；已认证端点还要
求现有 CSRF 与最近认证策略。

## 后果

- 注册多一步邮件往返，但消除立即会话、邮箱枚举和预劫持组合风险；
- 邮件投递成为身份可用性的关键依赖，必须监控延迟、失败率和积压，但 Provider 故障不得产生
  半激活账户或重复 Workspace；
- 账户启用 MFA 后，丢失全部第二因素不能仅凭邮箱恢复，这是有意的失败关闭策略；
- Outbox 引入新的加密密钥生命周期、迁移和备份恢复要求；删除旧密钥前必须确认没有未完成
  payload 或仍需恢复的备份引用；
- L1-002D/L1-002E 必须分别交付迁移、OpenAPI、真实 PostgreSQL/Redis 测试、威胁模型增量和
  独立安全复核；L1-006 Web 流程不得自行改变本 ADR 的认证语义。
