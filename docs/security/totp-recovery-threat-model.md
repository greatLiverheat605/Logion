# TOTP 与恢复码威胁模型

- 工作包：`L1-002B`
- 追踪：Issue #19
- 上游：PR #16、PR #18
- 状态：实现中，合并前必须由非实现者独立复核

## 范围与安全目标

本增量覆盖 TOTP 登记、密码后二次验证、一次性恢复码、恢复码重建和 TOTP 停用。邮件验证、邮件恢复、人工客服恢复、最终 Web UI 与生产密钥托管不在本工作包范围内。

安全目标是：密码验证不能绕过已启用的 TOTP；TOTP secret、恢复码和 MFA challenge 不以明文落库或进入日志；挑战、TOTP 时间步和恢复码均不可成功重放；认证方式变更需要最近登录、可信 Origin、CSRF、限流和审计。

## 资产与信任边界

| 资产          | 存储/传输                                   | 边界与要求                                                          |
| ------------- | ------------------------------------------- | ------------------------------------------------------------------- |
| TOTP secret   | 仅登记响应中短暂明文；数据库为 AES-GCM 密文 | 浏览器不得长期保存；响应 `no-store`；日志与导出禁止包含             |
| KEK           | 环境/外部 secrets 管理                      | 不进入仓库、镜像、数据库或 API；32 字节；带版本 ID                  |
| 随机数据密钥  | 每个账户独立生成并由 KEK 包装               | AES-GCM AAD 绑定用户和 key ID                                       |
| 恢复码        | 仅生成响应中显示一次                        | 数据库保存 HMAC 定位值和 Argon2id 慢哈希；单次使用                  |
| MFA challenge | 仅密码验证后签发                            | 随机 bearer token 只存 HMAC；短期、用途绑定、最多五次失败、单次使用 |

## 威胁、控制与验证

| 威胁                                 | 控制                                                                                            | 验证证据                                                                |
| ------------------------------------ | ----------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------- |
| 数据库泄漏暴露 TOTP secret           | 随机 DEK 加密 secret，版本化 KEK 包装 DEK；两个 AES-GCM 层分别使用 96 位随机 nonce 和 AAD       | 单元测试验证往返、明文不在密文中、篡改失败关闭                          |
| 只有密码即可登录已启用 TOTP 的账户   | 密码成功后仅签发 MFA challenge，不创建会话或 Cookie                                             | PostgreSQL/Redis 集成测试检查 `202`、无 access Cookie，二次验证后才签发 |
| MFA challenge 猜测、重放或暴力尝试   | 384 位随机 token、HMAC 落库、五分钟过期、行锁、成功即消费、五次失败即消费、Redis 限流           | 集成测试验证错误尝试、成功后重放拒绝                                    |
| 同一 TOTP 在多个请求中重放           | 记录最后成功时间步；验证凭据行 `FOR UPDATE`；只接受更大的时间步；允许前后各一个时间步的时钟漂移 | 集成测试使用相邻时间步并验证挑战重放；独立 reviewer 需补并发竞争测试    |
| 恢复码数据库撞库                     | 每码 16 个受限字符；HMAC 仅用于索引，最终由 Argon2id 验证                                       | 单元测试验证慢哈希与错误码拒绝；依赖审计验证 Argon2 实现                |
| 恢复码重复使用或重建后旧码有效       | 使用时对行加锁并原子写 `used_at`；重建时撤销所有未使用旧码                                      | 集成测试验证已用码拒绝、剩余数递减和新批次生成                          |
| 攻击者更改或停用 TOTP                | 需要已认证且最近创建的会话、可信 Origin、双提交 CSRF、当前未重放 TOTP 和独立限流                | 路由负向测试与人工安全复核                                              |
| 密钥缺失、版本错误或密文篡改导致绕过 | 解密错误统一失败关闭为可重试 `503`，绝不降级为密码单因子                                        | 单元测试篡改密文；生产配置拒绝开发 key ID                               |
| 验证码、secret 或恢复码进入日志/错误 | 请求验证错误只返回字段位置/类型；审计仅记录方法、结果和计数；响应不回显输入                     | 现有敏感输入错误测试与 OpenAPI 审查                                     |

## 密钥轮换与恢复

`LOGION_TOTP_ENCRYPTION_KEYS` 是 `key_id -> base64url(32 bytes)` 的版本化 keyring，`LOGION_TOTP_ACTIVE_ENCRYPTION_KEY_ID` 选择新写入使用的 KEK。轮换时先增加新 key 并切换 active ID，再由受控后台任务重新包装 DEK；确认所有记录迁移和备份保留期结束后才能删除旧 key。删除仍被记录或备份引用的 key 会使 TOTP 失败关闭，不能自动降级认证。

生产环境的 keyring 必须来自云 secrets/KMS 注入。当前 Compose 仅定义注入接口，不构成生产密钥托管方案。

## 迁移、回退与残余风险

- `0004_totp` 的空表回退会删除 TOTP、恢复码和未完成挑战；一旦存在真实认证数据，只允许前向修复，不得直接 downgrade。
- TOTP 抵抗钓鱼能力弱于 Passkey；产品应优先推荐 Passkey，并清楚标识密码 + TOTP 的残余钓鱼风险。
- 允许一个时间步的时钟漂移会扩大约 30 秒接受窗口，但最后时间步和行锁阻止成功重放。
- 已展示的恢复码可能被截图、剪贴板、恶意扩展或设备备份获取；UI 必须提供一次性展示、明确保存提示和重新生成入口。
- 当前没有邮件或人工恢复。用户同时丢失 Passkey、TOTP 设备和恢复码时，系统不会提供绕过认证的支持通道。
- 独立 reviewer 必须重点尝试：并发 TOTP/恢复码重放、挑战失败上限、旧 key 解密、跨用户密文替换、事务部分失败、日志泄漏和迁移回退。
