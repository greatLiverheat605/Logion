# 邮箱验证与账户恢复威胁模型

- 工作包：`L1-002C`、`L1-002D`、`L1-002E`
- 状态：邮箱验证、MFA 感知密码找回和安全通知实现增量
- 决策基线：`ADR-0002`、`ADR-0006`

## 范围与安全目标

本基线覆盖公开注册的邮箱所有权证明、初始密码设置、密码找回、MFA 恢复边界、安全通知和
事务性邮件投递。具体邮件 Provider、人工客服恢复、邮箱地址变更和最终 Web 视觉设计不在本
工作包内。

安全目标是：未知邮箱与已有邮箱的外部行为不可区分；验证前攻击者不能预置生效密码；邮件
token 单次、限时、用途绑定且不以明文落库；邮箱不能单独绕过已启用的 MFA；安全动作和邮件
投递失败不会产生部分提交。

## 资产与信任边界

| 资产             | 边界与要求                                                                  |
| ---------------- | --------------------------------------------------------------------------- |
| 验证/找回 token  | 浏览器 fragment 和确认请求中短暂明文；数据库只保存 HMAC；日志和审计禁止记录 |
| 初始/新密码      | 仅 TLS 请求中出现，校验后直接 Argon2id；不得进入 Outbox、错误或分析         |
| 收件地址         | 用户表中的个人数据；Outbox 副本必须加密并在终态清除                         |
| 投递加密密钥     | 独立版本化 keyring，由云 secrets/KMS 注入；不复用 TOTP 或 Session HMAC key  |
| Session/设备/MFA | PostgreSQL 为权威；找回成功原子撤销 Session，不能由邮件 Provider 改写       |
| 邮件 Provider    | 外部不可信传输方；只获得最小模板数据，不获得密码、MFA secret 或业务正文     |

## 威胁、控制与验证要求

| 威胁                               | 策略控制                                                                    | 后续验证证据                          |
| ---------------------------------- | --------------------------------------------------------------------------- | ------------------------------------- |
| 注册/找回枚举账户                  | 所有邮箱统一 `202`；相同响应体与延迟预算；IP 与 HMAC 邮箱双桶限流           | 存在/不存在/暂停/已验证响应对比测试   |
| 攻击者预注册受害者邮箱             | 注册阶段不设置生效密码；邮箱确认时才原子设置初始密码                        | 预注册后旧输入密码永远不能登录        |
| 邮件扫描器自动激活账户             | GET 不改变状态；token 在 fragment；确认只能通过可信 Origin POST             | GET、预取和 Referer 测试              |
| 数据库或日志泄漏 bearer token      | 随机 token 只存域分离 HMAC；Outbox payload 认证加密；错误和审计使用允许列表 | 数据库、日志、OpenAPI 和序列化检查    |
| token 重放、并发消费或用途混淆     | purpose/user/AAD 绑定、过期和撤销、行锁、单次 `used_at`、每用途单一活动流程 | 并发双消费、跨用途和旧 token 测试     |
| 邮箱接管绕过 TOTP                  | 已启用 TOTP 时，找回完成还需未重放 TOTP 或恢复码；无客服单人绕过            | 邮件 token 单独完成应被拒绝           |
| 找回后旧会话继续访问               | 密码替换和全 Session/Refresh 撤销处于同一事务                               | 旧 Access、Refresh 和并发刷新负测     |
| 邮件重试造成重复动作               | Outbox ID 作为幂等键；邮件只承载 token，权威消费仍在数据库                  | 重复、乱序、超时和 Provider 重试测试  |
| Outbox 泄漏收件地址/token          | 独立 keyring 认证加密、AAD 绑定、终态清除、死信脱敏、最小回执               | 密文篡改、跨行替换、旧 key 和日志检查 |
| 邀请接受绕过邮箱证明               | 邀请邮箱匹配之外强制 `email_verified_at`；确认与邀请接受是独立事务          | 未验证受邀者拒绝、验证后成功测试      |
| 投递故障导致半激活或重复 Workspace | 业务事务只写流程和 Outbox；确认事务原子创建密码/验证时间/个人 Workspace     | 故障注入和唯一 Workspace 不变量测试   |
| 旧客户端继续调用立即会话注册       | 新端点并行发布；生产禁止 legacy 注册创建 Session；兼容窗口后再版本化移除    | 生产配置测试和旧客户端响应测试        |

## 恢复能力矩阵

| 可用证明                            | 允许路径                              | 不允许路径                               |
| ----------------------------------- | ------------------------------------- | ---------------------------------------- |
| 已验证邮箱，无 MFA                  | 邮件 token 重置密码并撤销全部 Session | 保留旧 Session 或自动登录                |
| 已验证邮箱 + TOTP/恢复码            | 邮件 token + 未重放第二因素重置密码   | 仅凭邮箱关闭/绕过 MFA                    |
| 有 Passkey 或最近认证 Session       | 登录后走最近认证的密码变更            | 把 Passkey assertion 发送到邮件 Provider |
| 丢失邮箱但仍有 Passkey              | 登录并通过后续受控邮箱变更流程        | 客服直接替换邮箱                         |
| 邮箱、Passkey、TOTP、恢复码全部丢失 | 当前无自动恢复                        | 单人客服或仅凭个人资料绕过               |

## 残余风险与发布门槛

- 邮箱账户本身被接管时，无 MFA 用户仍可能被重置密码；产品应优先推荐 Passkey 和 TOTP。
- 邮件系统可观察收件人、时间和模板类型；不得在主题或模板变量中加入学习、研究或 Workspace
  正文。
- 统一响应不能完全消除基于收件箱观察的枚举；限流、安全通知和反滥用监控仍然必要。
- 邮件延迟会阻塞新用户激活。生产发布前必须确定 Provider、退信/投诉处理、SPF/DKIM/DMARC、
  数据区域和保留策略。
- 任何实现 PR 必须由非实现者独立复核，并完成真实 PostgreSQL/Redis、迁移往返、并发消费、
  Token/日志泄漏和恢复后 Session 失效测试。
- `L1-002D` 只建立加密事务性 Outbox，不包含真实 Provider 适配器。投递 Worker 完成前，新流程
  只能通过受控开发适配器和集成测试验证，不能宣称生产邮件可用。
- `L1-002E` 在恢复成功或第二因素连续失败达到上限时写入无 token 的加密安全通知；当前只保证
  事务性入队，真实送达、退信告警和 Provider 级幂等仍依赖后续投递 Worker。
