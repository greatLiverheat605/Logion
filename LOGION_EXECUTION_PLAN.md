## 1. 命名与定位审查

### 1.1 名称结论

正式名称采用 **Logion**，产品说明语为：**让学习、研究与长期成长形成可验证的闭环。**

`Logion` 简短、易形成视觉识别，含“言说、论述、知识表达”的语义联想，能覆盖学习者、研究者和知识工作者，不再绑定某一专业或单一用户。名称符合品牌延展要求，但上线前仍必须完成商标、主要域名、应用商店同名和社交账号可用性检索；未完成检索前不得宣称名称已获法律保护。

统一命名规则：

- 用户可见品牌只使用 `Logion`，不再出现“研途 Lab”“个人学习网站”；
- 仓库、包、域名占位与技术前缀使用小写 `logion`；
- 产品文档使用 `LOGION_<SUBJECT>.md`；
- Web 路由、文件夹和静态资源使用小写 kebab-case；
- 数据库使用 snake_case，类型使用 PascalCase，变量和 JSON 字段使用 snake_case；
- 用户内容可使用任何语言，不用文件名承载用户身份或敏感信息。

### 1.2 新定位

Logion 不是单一用户的个人待办工具，也不是课程内容平台。它是面向公众注册、支持个人和团队工作区、离线优先的学习与研究操作系统，将以下对象连接为闭环：

`目标 → 路线 → 任务 → 学习会话 → 笔记/产出 → 证据 → 验收 → 掌握度 → 复习 → 审查 → 调整`

核心承诺：

1. 用户能知道现在最值得做什么以及原因；
2. 每项完成都有产出或证据，而不只是勾选；
3. 离线、弱网、换设备和服务迁移时数据仍可用；
4. AI 是可选、可替换、可审计的增强层；
5. 从个人起步，可平滑扩展到导师、学习小组和研究团队；
6. 用户可以导出、删除、备份和迁移自己的数据。

### 1.3 面向所有用户的适配策略

“所有用户”不解释为一次性满足所有细分行业，而解释为：不把产品模型、默认内容或权限写死为某个特定人的经历，并满足广泛设备、能力和使用成熟度差异。

首发服务四类角色：

| 用户类型 | 首要目标 | 默认入口 | 必须成立的价值 |
|---|---|---|---|
| 自主学习者 | 建立路线并坚持执行 | 今日 | 清楚下一步、低成本记录、按时复习 |
| 学生/备考者 | 管理课程、作业、错题和阶段门槛 | 计划 | 任务与知识点、证据和测验连通 |
| 研究者 | 管理论文、问题、实验和研究证据 | 研究 | 研究活动与日常学习可追溯 |
| 导师/小组 | 查看被授权进度、反馈和协作 | 工作区 | 最小权限、可撤销分享、成员隔离 |

渐进式体验：新用户可用 3 分钟创建首个目标；普通用户无需理解 CRDT、Provider 或数据模型；高级设置仅在需要时显示；专业模板可以加载，但不改变通用核心模型。

## 2. 产品原则与不可变业务规则

1. **数据寿命高于界面寿命**：结构化数据、Markdown、稳定 ID、版本化迁移和开放导出。
2. **核心功能不依赖 AI**：AI 故障时计划、执行、记录、复习、研究、导出均可运行。
3. **证据优先**：活动、产出、验收和掌握度是不同状态；“完成”不自动等于“掌握”。
4. **离线优先**：本地写入先成功，再通过 Outbox 同步；不得因网络失败撤销已完成的本地操作。
5. **冲突不静默覆盖**：可安全合并的字段自动合并，关键状态和结构冲突必须保留双方版本并由用户确认。
6. **租户隔离**：所有业务对象属于 workspace；服务端不能信任客户端传入的 workspace 授权结论。
7. **AI 只产草稿**：AI 不得直接修改正式业务记录、掌握度、验收状态或权限。
8. **密钥不进入浏览器**：Provider 密钥和恢复秘密仅在服务端加密保存，日志、导出、前端响应不得泄露。
9. **可追溯、可恢复、可迁移**：关键变更有审计，备份可演练，导出可独立阅读。
10. **无障碍和响应式是完成条件**：不是发布后的优化项。

## 3. 信息架构与产品页面

### 3.1 公共站点

- 首页：一句价值主张、三步工作方式、适用人群、隐私与离线承诺、产品截图、价格入口、FAQ、注册 CTA；
- 模板库：通用学习计划、课程、研究、阅读、考试与 Agent 安全示例模板；
- 功能页：学习闭环、研究工作台、离线与数据主权、AI 能力；
- 定价页：免费个人版、个人专业版、团队版；最终金额由商业决策确定；
- 文档与更新日志：帮助、导入导出、隐私、安全白皮书和版本变化；
- 登录、注册、找回、邀请接受和法律页面。

公共页面必须支持 SEO 元数据、Open Graph、语义化 HTML、可分享稳定 URL、匿名浏览和不依赖登录的产品理解。

### 3.2 应用站点

主导航固定为：

1. **今日**：推荐任务、到期复习、阻塞、快速记录、同步状态；
2. **计划**：目标、路线、阶段、里程碑、知识点依赖、任务排期；
3. **记录**：学习会话、笔记、证据、验收、错题；
4. **资料**：链接、课程、论文、PDF 索引、附件与标签；
5. **研究**：论文卡片、研究问题、实验、运行、指标和结论；
6. **审查**：每日/每周/阶段审查、掌握度、复习队列、风险发现；
7. **AI**：Provider、模型、任务路由、草稿审批、用量与成本；
8. **工作区设置**：成员、角色、设备、安全、同步、导入导出、备份、账单。

移动端使用底部五入口（今日、计划、记录、审查、更多），不得删减开始会话、记录笔记、提交证据、复习和冲突处理等核心流程。

### 3.3 关键用户旅程

**首次使用**：注册/接受邀请 → 建立或加入工作区 → 选择“从空白、使用模板、导入文件” → 设置目标和每周投入 → 生成首个阶段 → 进入今日页。

**每日学习**：查看推荐及原因 → 开始会话 → 打开关联资料 → 记录理解与问题 → 提交证据 → 按标准验收 → 安排复习 → 完成日审查。

**离线使用**：打开已安装 PWA → 本地解锁 → 编辑已同步数据 → 附件进入待上传 → 联网后推拉同步 → 只对不可自动解决项提示冲突。

**研究工作**：登记论文 → 建立卡片与关联知识点 → 提出可证伪问题 → 设计实验 → 登记代码/数据/指标 → 记录失败与下一步 → 周审查评估推进。

**AI 辅助**：选择对象和字段 → 显示将发送的内容、Provider 与预算 → 用户确认 → 生成结构化草稿 → 比较差异 → 接受、编辑或拒绝 → 保留来源和审计。

**协作与分享**：所有者邀请成员 → 指定 workspace 角色 → 成员只见获授权空间 → 生成可撤销的只读快照/报告链接 → 到期或主动撤销后立即失效。

## 5. 核心对象与状态

顶级隔离链：`User ↔ WorkspaceMembership ↔ Workspace → Domain Objects`。一个用户可拥有或加入多个 workspace；一个 workspace 可有多个成员；每项查询和变更均校验 membership 与角色。

核心对象：User、Workspace、Membership、Invitation、Device、LearningPlan、PlanVersion、Phase、Milestone、Topic、TopicDependency、Task、StudySession、SessionEvent、Note、Resource、Attachment、EvidenceItem、VerificationRecord、MasteryRecord、ReviewSchedule、QuizItem、QuizAttempt、ErrorPattern、AuditReview、ReviewFinding、Paper、ResearchQuestion、Experiment、ExperimentRun、ExperimentMetric、AIProvider、AIModel、AITaskRoute、AIRun、AIOutputDraft、SyncOperation、SyncConflict、AuditEvent、Tag、ShareSnapshot、Subscription。

关键状态机：

- Task：`backlog → planned → in_progress → submitted → verified → done`，另有 `blocked/cancelled`；缺少证据时不得自动进入 verified/done；
- Verification：`pending/passed/failed/needs_revision`；
- Review：`scheduled/due/in_progress/completed/skipped`；
- AI Draft：`draft → accepted/rejected/expired`；接受时校验目标版本；
- Attachment：`local_only/pending_upload/uploading/verified/failed/deleted`；只有 verified 可作为完成证据；
- Conflict：`open/resolved_local/resolved_remote/resolved_merge/dismissed`；
- Membership：`invited/active/suspended/revoked`。

所有同步业务实体至少含：`id(UUIDv7)、workspace_id、version、created_at、updated_at、deleted_at、created_by、updated_by`。事件表使用 workspace 内单调 `sequence`。时间在服务端存 UTC `timestamptz`，客户端按 locale/timezone 展示。

## 6. 总体技术方案

### 6.1 推荐架构

- Web：TypeScript + React + Next.js（App Router）构建公共站与应用壳；核心应用为可安装 PWA；
- 本地数据：IndexedDB + Dexie（或经 ADR 选择等价库），Outbox 与同步状态和业务数据同事务提交；
- 笔记：Markdown 可读快照 + Yjs 更新；
- API：Python 3.12+、FastAPI、Pydantic、SQLAlchemy 2、Alembic；
- 数据：PostgreSQL；Redis 仅用于队列/限流/短期缓存，不作为权威数据源；
- 后台任务：轻量 worker 处理附件校验、备份、AI 异步任务和 URL 检查；
- 文件：首发本地/挂载卷，接口兼容 S3 对象存储；
- 部署：Docker Compose 起步，反向代理终止 TLS；公共站、API、worker、PostgreSQL、Redis、备份任务分服务；
- 契约：`/api/v1`、OpenAPI 3.1、`sync-v1`，错误均带机器码、可理解说明、`retryable` 与 `request_id`。

### 6.2 离线与同步摘要

客户端先本地事务写业务对象和 Outbox。Push 按 operation_id 幂等，服务端验证设备、membership、workspace、实体版本和 payload hash；成功后写业务数据、processed_operation、change_log、audit_event。Pull 使用 workspace 单调 cursor，分页且可断点。首次同步使用版本化快照并在本地完成校验后原子切换。

Markdown/Yjs 更新按文档流合并并定期压缩，同时保留可读版本；关键状态、层级移动、删除与验收冲突不得用最后写入获胜。服务器从备份恢复后提升 sync epoch，旧设备必须重新 bootstrap，防止旧 Outbox 污染恢复数据。

### 6.3 安全摘要

HTTPS、Argon2id、WebAuthn、TOTP、一次性恢复码、短期 access token 与轮换 refresh token；Cookie 使用 HttpOnly/Secure/SameSite；写操作防 CSRF；Markdown 白名单渲染；自定义 AI Base URL 与 URL 健康检查防 SSRF/DNS rebinding；附件校验 MIME、扩展、大小、SHA-256 与隔离路径；所有对象访问做跨 workspace 负向测试；敏感密钥采用信封加密并从日志、导出和前端响应中剔除。

## 7. 推广性与增长设计

推广建立在可证明价值而非社交噪音上：

1. 用户完成首周后生成隐私安全的“学习进展卡”，默认不含笔记正文、真实姓名或敏感研究内容；
2. 用户可分享模板、只读成果快照或公开方法论页，默认私密、显式开启、随时撤销；
3. 分享页带模板预览与“复制到我的 Logion”入口，形成 `成果 → 模板 → 新用户 → 新成果` 循环；
4. 邀请导师/同伴查看指定工作区或报告，权限最小化并可设有效期；
5. 公共页面提供真实可核验的功能、安全和性能描述，不使用无法证明的用户量、合规或“零风险”声明；
6. 产品分析采用最小化、可关闭、汇总式事件；不采集笔记正文、AI 密钥和研究内容。

北极星指标为“每周完成至少一次有证据的学习闭环的活跃 workspace 数”。辅助指标包括激活率、首个证据闭环时间、四周留存、复习按期率、模板复制转化和邀请接受率。不得用无意义打卡数驱动产品。

## 8. 并行研发计划

### 8.1 工作流与依赖

| 流 | 工作内容 | 可并行起点 | 合并门槛 |
|---|---|---|---|
| A 产品/设计 | 设计系统、公共站、应用信息架构、移动与空/错/离线态 | 立即 | 原型评审、可访问性走查 |
| B 平台/身份 | 工程骨架、注册、workspace、RBAC、设备、安全 | 规格冻结后 | 越权负测、会话撤销通过 |
| C 数据/同步 | 数据库、迁移、IndexedDB、Outbox、push/pull、冲突 | 与 B 并行 | 双设备离线总场景通过 |
| D 学习核心 | 计划、任务、会话、笔记、证据、验收 | B/C 契约稳定后 | 单日闭环与导入通过 |
| E 复习/研究 | 掌握、复习、审查、论文、问题、实验 | D 基础对象后 | 复习回流与研究导出通过 |
| F AI | Provider、路由、预算、草稿、审批 | 身份和审计接口后 | 密钥扫描、草稿隔离通过 |
| G 增长/协作 | 公共站、模板、邀请、只读分享、分析 | A/B 可并行 | 隐私审查与撤销测试通过 |
| H 质量/运维 | CI、测试、监控、备份、恢复、发布 | 从第一天持续 | 发布门槛全部通过 |

### 8.2 阶段依赖摘要

正式日历排期、并行能力和风险缓冲以第 21.6 节为准。本节只定义不可颠倒的阶段关系：

1. Phase 0 工程实施准备：基线、ADR、仓库、契约生成和 CI/CD；
2. Phase 1 可信身份底座：身份、Workspace/Space、权限、设备和审计；
3. Phase 2 离线同步底座：本地库、Outbox、bootstrap/push/pull、冲突和附件队列；
4. Phase 3 通用学习闭环：用户自定义目标、任务、会话、内容、证据和验收；
5. Phase 4 学习科学与四类场景：复习审查及备考、自学、研究、导师/小组闭环；
6. Phase 5 AI、分享与数据主权：Provider 草稿、模板、分享、搜索通知、导入导出和备份；
7. Phase 6 发布加固：性能、安全、无障碍、兼容、恢复、监控、灰度和运行手册。

Phase 1 与 Phase 2 可在身份/权限和同步契约冻结后部分并行；Phase 4 的场景包可在 Phase 3 公共对象稳定后并行。任何并行都不得跳过前置契约、独立复核或退出条件。

## 9. 测试与发布验收

### 9.1 质量目标

- 核心领域单元测试覆盖率 ≥ 85%，全仓覆盖率不得通过排除关键代码虚增；
- 在线非 AI 普通 API p95 < 500 ms；常用本地读取在中等设备上近似即时；
- 基线容量：10 万任务、100 万事件、5 万笔记/资料、1 万附件、5 千论文、10 万 AI 运行；
- 最新两个稳定版 Chrome/Edge/Firefox、当前 Safari；移动 Chrome 与 iOS Safari/PWA 明确验证和记录降级；
- WCAG 2.2 AA 自动检查 + 键盘/读屏人工关键流检查；状态不只依赖颜色，触控目标、焦点和错误提示可感知。

### 9.2 发布总场景

1. 两名不同用户注册并分别创建 workspace，交叉访问资源全部被拒绝；
2. 所有者邀请 Viewer，Viewer 只能查看授权内容，撤销后会话即时失效；
3. 新用户从通用模板创建计划并完成任务、证据、验收、复习和日审查；
4. 导入原 47 天计划，阶段、任务、资料占位、门槛和日期正确；
5. 电脑和手机首次同步后同时断网；分别修改同一笔记不同段、同一验收状态和不同对象；
6. 联网后笔记安全合并，状态冲突要求选择，操作重试不重复，附件校验后上传；
7. 丢失设备被撤销，本地残留风险得到明确提示，服务端令牌不可继续使用；
8. 一篇论文、一个研究问题、一次实验及指标可关联并导出 BibTeX/JSON/CSV/Markdown；
9. 两个 Provider 配置完成，首选失败按策略降级；敏感内容发送前确认；拒绝草稿不改正式记录；
10. URL 检查不能访问回环、私网和云元数据；恶意 Markdown、附件和跨 workspace ID 被拒绝；
11. 自动备份完成，在空环境恢复后业务对象、附件、游标、审计和租户关系完整；旧设备被要求重新 bootstrap；
12. 公共首页、注册、应用关键流在手机、键盘和读屏路径可用，SEO/分享卡信息正确；
13. 用户导出全部数据后删除账户；保留/删除周期、共享链接和审计处理符合隐私说明。

任何 P0/P1 缺陷、数据丢失、静默冲突、租户越权、密钥泄漏或备份不可恢复均阻止发布。

## 13. 本轮重新审查结论与修正清单

### 13.1 符合项

现有基线已经正确覆盖：Logion 品牌、多租户 workspace、公众注册、离线优先、多设备同步、任务证据、复习审查、研究对象、可配置 AI、数据导出、备份恢复、公共营销站、成员邀请、发布质量和 AI 开发纪律。旧版“仅个人使用”的限制已经被正确废止并归档。

### 13.2 原文档仍存在的产品缺口

本轮将下列缺口提升为正式要求：

1. 只有宽泛用户分类，缺少备考、自学、研究生三类场景的专用对象和验收闭环；
2. RBAC 只有 workspace 粒度，缺少私人空间、共享空间、对象级分享和字段级快照边界；
3. 缺少考试日期、考试大纲、模考、题库、科目权重、成绩趋势等备考对象；
4. 缺少课程/专题、收件箱、学习项目和成果集等普通自学对象；
5. 缺少文献综述矩阵、研究声明证据、实验对比组、导师反馈关闭记录等研究生深层能力；
6. 缺少统一搜索、通知、日历订阅、邮件任务和离线通知的产品定义；
7. 缺少模板版本、来源、许可证、审核、复制升级和恶意模板治理；
8. 缺少 SaaS 运营后台、滥用处置、配额、套餐权益、支付状态和客户支持边界；
9. 缺少公共注册后的年龄/隐私、数据保留、内容版权和产品分析约束；
10. 缺少市场对标与明确的“不做什么”，容易退化为 Notion、Anki 或 LMS 的功能堆叠；
11. 原型虽覆盖主要页面，但还需要证明多用户、不同人群、搜索通知、考试场景和权限差异；
12. “所有用户”容易被误读为无限范围，需要明确首发人群与扩展模型。

### 13.3 修正后的首发边界

首发“适配所有用户”具体定义为：平台核心模型不绑定专业，且对**备考者、自主学习者、研究生/研究者、导师或学习小组**四类人群提供可完成的主流程。企业培训、K-12 学校全量教务、MOOC 视频托管、公共社交社区不属于首发目标。

产品必须允许一个用户同时拥有：

- 私人 workspace：默认不可被任何其他用户看到；
- 共享 workspace：学习小组、导师课题、课程或团队；
- 多个 profile track：备考、自学、研究可同时启用，而不是注册时永久选定一种身份。

### 14.1 竞争定位

Logion 的定位不是“另一个笔记/任务/卡片工具”，而是：

> **面向长期学习与研究的证据驱动操作系统：把计划执行、知识掌握、复习审查、研究实验和协作反馈连接起来，并提供离线优先的数据主权与可替换 AI。**

### 14.2 必须守住的差异

1. **证据闭环**：完成、验收、掌握是三个不同状态；
2. **跨周期**：从一次考试到研究生课题，数据无需迁移到另一套系统；
3. **离线优先**：不是只在服务器有一份数据；
4. **研究原生**：论文、问题、实验、指标和失败记录是一等对象；
5. **AI 可替换**：BYOK/自定义 Provider、明确发送范围、输出先草稿；
6. **协作但默认私密**：共享来自显式授权，不把个人学习日志默认暴露给团队；
7. **强核心、弱模板耦合**：模板改变起点，不改变底层不变量。

### 14.3 竞争禁区

- 不用“支持所有学习方式”掩盖首发闭环不完整；
- 不以连续打卡、积分和排行榜替代真实成果；
- 不复制 Zotero 的完整引用引擎、Moodle 的全部教务、Notion 的无限数据库或 Anki 的全部高级制卡；
- 不承诺“AI 自动学会你的全部知识”“零幻觉”“绝对安全”；
- 不以未经验证的用户量、学校合作、认证或性能数字推广。

## 15. 用户分层与专用功能包

### 15.1 共同核心

所有用户共享：目标、计划、阶段、知识点、任务、资料、会话、笔记、证据、验收、掌握、复习、审查、搜索、通知、导出、同步和 AI 草稿。

专用功能以 workspace 模块开关启用，不创建互不兼容的四套产品。

### 15.2 备考与课程模块

核心对象：`Exam、Subject、SyllabusNode、QuestionBank、Question、PracticeSet、Attempt、MockExam、ScoreRecord、ErrorPattern、ExamCountdown`。

必须支持：

- 考试日期、目标分、科目及权重；
- 大纲覆盖率与重要度；
- 历年题/模拟题索引；
- 题目、尝试、答案、耗时、信心和错因；
- 错题回流知识点和复习；
- 模考时间约束、成绩趋势和薄弱项；
- 考前阶段负荷与缓冲日；
- 教师/同伴只看用户明确共享的成绩或报告。

不在首发内建设题库内容交易市场；用户导入内容需确认版权和来源。

### 15.3 自主学习模块

核心对象：`LearningGoal、LearningTrack、CourseIndex、Project、Deliverable、InboxItem、PortfolioItem`。

必须支持：

- 资料/想法快速进入收件箱；
- 目标拆成路线和成果；
- 课程章节与知识点映射；
- 学习项目、里程碑和作品证据；
- 每周可用时间和负荷平衡；
- 中断后重新规划而不伪造连续性；
- 公开成果快照与模板复制。

### 15.4 研究生与研究模块

在现有 Paper/ResearchQuestion/Experiment 基础上增加：

- `Claim`：研究声明及其支持/反对证据；
- `LiteratureMatrix`：论文在问题、数据、方法、结果和局限上的对比；
- `DatasetRecord`：来源、许可、版本、划分和敏感性；
- `Protocol`：实验步骤和预注册信息；
- `Baseline`、`Ablation`、`RunGroup`：实验公平比较；
- `SupervisorFeedback`：导师反馈、责任人、截止日期、关闭证据；
- `ManuscriptSection`：章节、引用、待验证声明和写作状态；
- 研究时间线、会议截止日期和复现实验检查表。

### 15.5 导师、小组与课程协作模块

必须支持：

- 个人空间与共享空间严格分开；
- 作业/反馈/审阅任务；
- rubric 和验收标准；
- 成员进度只在授权范围聚合；
- 只读报告快照、有时效邀请和撤销；
- 导师反馈转为用户任务，但不能直接修改用户私人掌握度；
- 工作区活动流不显示私人笔记正文。

### 16.1 搜索

首发搜索同时覆盖标题、Markdown 可读快照、资源元数据、论文、任务和标签；按 workspace 隔离，并能在本地对已同步数据工作。敏感正文不发送第三方搜索服务。搜索结果必须显示对象类型、workspace、匹配片段和权限来源。

### 16.2 通知

通知分为：到期任务、到期复习、邀请/反馈、同步/冲突、安全、AI 完成、配额/账单。默认采用产品内通知；邮件和 Web Push 可配置。学习提醒允许安静时间、时区和频率上限；安全通知不可完全关闭。

### 16.3 日历

提供周/月视图与 iCalendar 只读订阅；计划任务、考试、模考、论文截止、审查和复习可以进入日历。首发不承诺与所有第三方日历双向写入。

### 16.4 模板

模板是版本化包，包含对象图、默认规则、资源占位、目标人群、适用版本、语言、作者、许可证和变更日志。复制后用户拥有独立实例；模板升级只能生成差异建议，不能覆盖用户内容。

## 17. 多用户、隐私与权限细化

### 17.1 空间模型

`Account → Workspaces → Spaces → Objects`：

- Workspace 是租户和计费边界；
- Space 是 workspace 内的可见性边界，可为 private/shared；
- 对象必须属于一个 Space；
- 个人 workspace 默认只有一个私有 Space；
- 共享 workspace 可以有成员私有草稿区和团队共享区；
- 跨 workspace 引用使用显式快照或外部引用，不共享原对象主键权限。

### 17.2 角色

基础角色：Owner、Admin、Editor、Contributor、Reviewer、Viewer。Member 仅作为兼容名称，不应成为含义模糊的最终权限。

角色只提供默认能力；分享快照、对象审阅和敏感 AI 发送使用额外能力检查。所有关键服务端操作依据当前 membership、space 可见性、对象归属和 entitlement 共同决策。

### 17.3 私密默认

- 新建个人内容默认 private；
- 加入团队不自动迁移个人内容；
- 分享报告默认排除笔记正文、附件、AI 输入和私人错题；
- 管理员不能因为是管理员就读取用户私人 Space；
- 团队聚合指标使用最小字段并显示统计范围；
- 退出 workspace 时允许导出本人可导出的内容，团队对象的所有权按规则处理。

### 17.4 公共注册与年龄

在完成未成年人合规评估前，首发默认面向达到当地数字服务最低年龄且可独立同意条款的用户；产品不得在未建立家长同意、数据最小化和删除流程前主动定位低龄儿童。具体年龄门槛由上线地区法律审查决定。

## 18. SaaS 运营、套餐与治理

### 18.1 套餐与 entitlement

套餐不能散落在前端判断。服务端使用 entitlement 决定：workspace 数、成员数、存储、历史保留、AI 用量、模板发布、分享有效期和备份能力。降级不得立即删除超额数据，应进入只读/宽限期并允许导出。

建议验证的套餐结构：

- Free：个人核心闭环、有限存储和基础模板；
- Pro：更多存储、高级审查、研究能力、BYOK AI 与历史；
- Team/Education：成员、角色、反馈、共享模板和管理能力。

正式价格必须经成本、支付渠道、目标市场和用户访谈验证，本计划不预设金额。

### 18.2 运营后台

独立的系统 Operator 控制面至少包含：用户/workspace 状态、配额、邀请滥用、邮件状态、备份、任务队列、Provider 故障、功能开关、版本、审计查询和数据删除工单。Operator 与 workspace Admin 是不同权限域；默认不可读取用户正文。

### 18.3 模板和公开内容治理

首发公共模板由官方审核发布。开放用户发布前必须增加举报、版权声明、恶意链接扫描、版本撤回、作者身份和审计。复制模板时不携带原作者私人数据、密钥、成员和外部访问 Token。

### 18.4 支持与事故

提供状态页、支持工单、数据导出/删除请求和安全报告入口。涉及账户恢复、数据恢复、支付和隐私的支持动作需要双重确认与审计；客服不能索要密码、TOTP、恢复码或 AI Key。

## 19. 原型深化验收范围

原型必须覆盖并可交互检查：

1. 公共首页、适用人群、市场差异、模板、定价占位、隐私和注册入口；
2. 身份选择不是永久分叉，用户可在备考/自学/研究/导师场景间切换；
3. Workspace 和 Space 切换，清晰显示私人/共享边界；
4. 今日页显示推荐理由、考试倒计时/研究截止、复习和证据；
5. 计划、学习记录、资料、研究、审查和 AI 中心；
6. 备考页面展示大纲、模考、错题、成绩趋势；
7. 全局搜索、通知、同步状态、冲突中心和快速新建；
8. 成员、角色、只读分享和撤销；
9. AI Provider、发送范围、预算和草稿审批；
10. 桌面与移动布局；键盘焦点、对话框和减少动效；
11. 空状态、离线状态、权限不足、配额、错误和恢复提示；
12. 原型数据覆盖不同学科，不能只展示 AI/Agent 安全。

原型不需要连接真实后端，但不能用交互暗示违反业务不变量，例如“一键完成即掌握”“离线时成功调用云 AI”“Viewer 可编辑”或“团队管理员可看私人笔记”。

## 20. 修订后的执行优先级

### P0：平台可信底座

身份、workspace/space、权限、数据库迁移、离线同步、附件、审计、导出、备份恢复、安全和可访问设计系统。

### P1：通用证据闭环

目标/计划/任务、会话、资料/笔记、证据/验收、知识/测验/复习/审查、搜索、通知、模板导入。

### P2：三类人群闭环

备考的大纲—练习—错题—模考；自学的路线—项目—成果；研究的论文—声明—问题—实验—反馈。

### P3：协作与推广

邀请、角色、Space、报告快照、模板复制、公共站、套餐 entitlement 和最小运营后台。

### P4：可配置 AI

Provider、模型、路由、字段选择、敏感策略、成本、异步任务、草稿差异和确认。

尽管 P4 排在功能顺序之后，AI 网络安全、密钥模型和草稿不变量必须在 P0 规格中冻结。开发可并行，但不得让 AI 成为 P1/P2 完成的前置依赖。

## 21. 代码实施总计划（版本 1.1）

### 21.1 当前阶段与开工结论

Logion **可以进入代码实施阶段**，但实施分成两个连续关口：

1. **现在进入 Phase 0：工程实施准备**。允许建立仓库骨架、开发环境、契约生成、测试框架、CI/CD、ADR 和最小健康检查；不允许提前堆叠业务页面或用演示数据冒充业务完成。
2. **Phase 0 退出评审通过后进入 Phase 1：业务纵向切片**。第一条业务链必须是“账户 → Workspace → Private Space → 创建自定义目标 → 本地保存 → 同步 → 换设备读取”，而不是先建设大量互不连通的页面。

用户类型只决定能力建议和入口排序，不决定或预装用户内容。备考、自学、研究、导师/小组均为可启停、可组合的 `profile_track`；“某项考试”“某课程”“某导师课题组”“某研究方向”等上下文必须由用户创建、导入或通过模板安装。模板只是版本化起点，安装后成为用户数据，绝不能在代码、迁移、种子数据或权限逻辑中写死。

### 21.2 单仓库模块地图

首发采用单仓库，以共享契约为中心。目录是稳定所有权边界；只有完成 ADR 才能更改顶层边界。

| 目录/模块 | 责任 | 允许依赖 | 不负责 |
|---|---|---|---|
| `apps/web` | Next.js 公共站、应用壳、路由、PWA 和各 feature UI | `packages/contracts`、`packages/ui`、`packages/offline`、`packages/config` | 服务端授权、权威业务状态 |
| `apps/api` | FastAPI 入口、应用用例、领域模块、持久化适配器 | 后端内部 domain/port、生成契约 | 长任务、浏览器本地状态 |
| `apps/worker` | AI、附件校验、导入导出、通知、备份编排等异步任务 | 与 API 相同的用例/端口 | 独立复制业务规则 |
| `packages/contracts` | OpenAPI 快照、生成的 TS 类型、错误码、事件与同步协议 schema | 无业务实现依赖 | 手写重复 DTO |
| `packages/ui` | 设计令牌、可访问基础组件、图表替代视图 | `packages/config` | 领域用例与数据请求 |
| `packages/offline` | IndexedDB schema、repository、Outbox、同步客户端、冲突状态机 | `packages/contracts` | 服务端冲突裁决、UI 页面 |
| `packages/config` | 共享 lint/type/test/build 配置 | 无业务模块 | 运行时秘密 |
| `infra` | 容器、反向代理、环境、部署、监控、备份恢复 | 已发布镜像和迁移产物 | 产品业务逻辑 |
| `tests` | 跨应用契约、E2E、安全、恢复、性能夹具 | 对外契约和运行环境 | 绕过模块公开接口 |
| `docs/adr` | 架构决策与替代方案 | 两份根目录基线 | 重复产品需求 |

`apps/api` 内部按领域语言拆分为 `identity`、`workspaces`、`planning`、`content`、`memory`、`exam`、`research`、`collaboration`、`ai_gateway`、`integrations`、`audit`。`apps/web` 使用同名 feature 边界。只有出现真实代码和测试时才创建目录，禁止预先生成大量空模块。

### 21.3 依赖顺序与契约所有权

```text
工程/配置/可观测性
        ↓
身份认证 → Workspace/Space/Permission → 审计
        ↓                     ↓
本地数据库/Outbox ← 同步协议/API → 服务端变更日志
        ↓
计划/任务 ── 内容/资料/附件 ── 会话/证据/验收
        ↓
掌握/复习/审查
        ↓
备考 | 自学 | 研究 | 导师协作（可并行场景包）
        ↓
AI 草稿 | 搜索/通知/日历 | 分享/模板 | 运营
        ↓
发布加固、迁移演练、备份恢复、灰度
```

共享契约实行单一负责人制度：

- OpenAPI、错误码、同步 operation、IndexedDB schema、权限名和数据库迁移序列各指定一名 `contract owner`；
- 其他工作包只能提交契约变更提案，不得与 owner 同时直接修改同一共享主文件；
- 契约合并顺序为“失败测试/兼容性说明 → 契约 → 生成产物 → 服务端 → 离线层 → UI”；
- 跨模块调用通过公开 use case、port 或事件，禁止跨目录直接访问数据库表或内部组件；
- 循环依赖、重复枚举、重复权限判断和手写前后端 DTO 均阻止合并。

### 21.5 CI/CD 分层

| 流水线 | 触发 | 目标时长 | 强制内容 | 结果 |
|---|---|---:|---|---|
| PR Fast | 每次 PR/更新 | ≤15 分钟 | 变更范围、格式、Ruff、Python 类型、ESLint、TS strict、单元测试、生成物漂移、密钥/依赖检查、受影响构建 | 必需检查；失败不得合并 |
| PR Integration | 命中 API/DB/offline/auth 等高风险路径 | ≤30 分钟 | PostgreSQL/Redis、Alembic upgrade/downgrade、OpenAPI breaking diff、契约、IndexedDB 升级、权限负测、受影响 E2E | 必需检查；按路径自动触发 |
| Main | 合并主分支 | ≤40 分钟 | 全量构建、集成、镜像、SBOM、镜像/IaC 扫描、签名、部署临时环境 smoke | 只发布不可变候选产物 |
| Nightly | 每晚 | 可长跑 | 全量浏览器 E2E、离线/乱序/重复、模糊/安全、可访问性、容量回归、依赖许可证 | 失败建立阻断问题，连续失败禁止 RC |
| Release Candidate | 版本标签/手动审批 | 可长跑 | staging 数据迁移、完整 E2E、安全/隐私/WCAG、备份与空环境恢复、旧客户端兼容、回滚/前向修复演练 | 生成发布证据包 |
| Production | 人工批准 | 分阶段 | 数据库 preflight、备份、5% 灰度、观测窗口、扩大至 25%/100%、自动健康判定 | 人可随时中止；不可由 AI 自动批准 |
| Scheduled Audit | 每周/月/季度 | 定期 | 依赖更新、密钥轮换提醒、备份可读/完整恢复、权限审计、灾难演练 | 报告留档并跟踪整改 |

CI 实现可以首选 GitHub Actions，但所有关键命令必须封装为仓库脚本或任务入口，使其可迁移到其他 CI。不得只在 CI YAML 中隐藏业务校验。

基线合计 **20–27 周**，另设 **20% 风险缓冲**，现实首个公开稳定版本为 **24–33 周**。若要更早验证市场，可在 Phase 3 后发布受控 Alpha（约 10–14 周），但不得宣称满足完整首发范围，也不得移除多租户、离线、认证和数据恢复门槛。任何固定日期都应在 Phase 0 完成容量校准后转成带负责人和实际开始日期的迭代计划。

### 21.7 Phase 0 开工清单与退出门槛

Phase 0 按以下顺序执行：

1. 批准本版两份根目录基线并冻结首批需求 ID；
2. 建立 `ADR-0001` 单仓/模块边界、`ADR-0002` 身份会话、`ADR-0003` 离线同步、`ADR-0004` Workspace/Space 权限、`ADR-0005` 部署与备份；
3. 创建最小目录、锁文件、统一格式/类型/测试命令和环境示例；
4. 创建 Web/API/worker 健康检查及 PostgreSQL/Redis 本地环境；
5. 建立 OpenAPI 生成、错误码和同步协议 schema 的单一生成链；
6. 建立 PR/main/nightly 流水线、分支保护、CODEOWNERS/等价所有权和合并队列；
7. 建立最小可观测性、迁移空库测试、SBOM、密钥扫描和依赖更新策略；
8. 用一个非业务“健康检查变更”验证失败门禁、候选镜像、临时环境和回滚路径；
9. 对后续 Phase 1 工作包做依赖和目录冲突检查，确定每个 AI/人员的允许修改范围。

仅当以上九项有可复现证据、两份基线无未解决冲突、主分支全绿且没有共享文件的并发 owner 时，Phase 0 才能标记完成。
