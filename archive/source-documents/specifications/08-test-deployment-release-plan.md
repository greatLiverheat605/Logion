# 研途 Lab 测试、部署与发布计划

> 目标：在真实长期学习数据进入系统前，证明核心业务、离线同步、安全和恢复能力达到可用门槛。

---

## 1. 质量原则

1. 先验证数据不丢，再优化体验；
2. 同步、认证、备份和迁移属于最高风险模块；
3. AI 不得阻塞核心测试；
4. 失败实验和缺陷必须形成回归测试；
5. 不以单元测试覆盖率代替端到端验证；
6. 发布前必须实际恢复备份；
7. 生产升级必须可停止、可观察、可恢复。

---

## 2. 测试层级

### 2.1 静态检查

后端：

- Ruff format/check；
- mypy 或 Pyright；
- 依赖漏洞扫描；
- secret scan；
- Alembic 多头迁移检测。

前端：

- TypeScript strict；
- ESLint；
- 构建检查；
- PWA manifest 与 Service Worker 检查；
- 依赖漏洞与 secret scan。

### 2.2 单元测试

重点：

- 状态机；
- 复习算法；
- 掌握度规则；
- 计划版本；
- 冲突字段判断；
- URL 安全分类；
- AI 路由；
- 成本估算；
- 密钥加解密；
- 附件状态机；
- Token 旋转和恢复码。

目标：核心领域模块语句/分支覆盖率不低于 85%，安全关键函数接近 100%。

### 2.3 数据库集成测试

使用真实 PostgreSQL 容器测试：

- 事务回滚；
- 唯一约束；
- 外键；
- 软删除；
- 同步操作原子性；
- 变更日志 sequence；
- 并发乐观锁；
- 全文索引；
- 迁移升级。

SQLite 不能替代 PostgreSQL 集成测试。

### 2.4 API 契约测试

- OpenAPI schema 校验；
- 请求/响应模型；
- 统一 Problem Details；
- 认证和权限；
- 幂等；
- 分页；
- 错误码；
- 敏感字段 writeOnly/不返回。

前端使用生成的客户端或契约 mock，防止接口漂移。

### 2.5 前端与本地数据库测试

- IndexedDB 事务；
- 本地迁移；
- Outbox 生成；
- 断网状态；
- Service Worker 更新；
- 本地解锁；
- 未同步警告；
- 浏览器配额不足；
- PWA 重启后恢复。

### 2.6 端到端测试

使用至少两种浏览器上下文模拟两台设备，并控制网络在线/离线。

## 4. 离线与同步测试

### 4.1 网络故障

- 请求发送前断网；
- 服务端处理成功但响应丢失；
- push 部分成功；
- pull 中断；
- 网络反复抖动；
- 长时间离线后大量操作；
- Token 过期后重认证。

### 4.2 并发冲突

- 不同字段自动合并；
- 同一任务状态冲突；
- 计划日期冲突；
- 删除与修改；
- 标签集合；
- 附件/证据追加；
- 笔记标题冲突与正文 CRDT 合并同时发生。

### 4.3 幂等

- 同一 operation 重复 2、10、100 次；
- 相同 ID 不同 payload；
- 同一附件完成请求重试；
- 同一 AI 运行重试；
- 同一 session event 重复。

### 4.4 快照

- 初次快照成功；
- 每页中断；
- 快照过期；
- 快照期间服务端继续写入；
- 快照完成后从固定 cursor 拉增量；
- 本地临时库原子切换。

### 4.5 恢复后的同步

- 服务端恢复旧备份；
- sync epoch 变化；
- 旧设备尝试 push；
- 强制重新 bootstrap；
- 未同步本地内容先导出或协调恢复。

## 6. 认证和安全测试

### 6.1 Passkey

- 正常注册和登录；
- challenge 重放；
- 错误 origin/RP ID；
- sign counter 异常；
- 删除最后恢复方式；
- 多 Passkey；
- 撤销设备。

### 6.2 密码/TOTP/恢复码

- Argon2id 参数检查；
- 错误密码和 TOTP 限流；
- TOTP 时间偏差窗口；
- 恢复码单次；
- 重新生成使旧码失效；
- 日志不包含 secret/code。

### 6.3 Token

- Refresh Token 旋转；
- 旧 Token 重用撤销家族；
- Cookie 属性；
- CSRF；
- 设备撤销后刷新失败；
- 并发刷新。

### 6.4 Web/API

- 存储型和反射型 XSS；
- Markdown 危险 URL；
- SQL 注入；
- IDOR/跨 workspace；
- SSRF；
- CORS；
- 错误堆栈泄漏；
- OpenAPI 文档暴露；
- 大请求和速率限制。

## 8. 性能与容量

### 数据集

生成长期规模：

- 100,000 任务；
- 1,000,000 状态/学习/审计事件；
- 50,000 笔记/资料；
- 5,000 论文；
- 10,000 附件元数据；
- 100,000 AI 运行。

### 指标

- 今日任务本地读取 < 100 ms 目标；
- 普通在线 API p95 < 500 ms；
- push 100 操作可接受时间 < 2 s 目标；
- pull 500 变更 < 2 s 目标；
- 首次快照可续传；
- 数据库备份时间和恢复时间有记录；
- CRDT 压缩不阻塞普通请求。

目标需要在选定云平台和前端实现后校准。

## 14. 发布流程

### 14.1 候选版本

1. 冻结 schema；
2. 生成版本说明；
3. 构建并扫描镜像；
4. 部署 staging；
5. 导入一份脱敏真实结构数据；
6. 执行迁移；
7. 执行 E2E 和恢复；
8. 签署发布检查单。

### 14.2 生产发布

1. 通知维护窗口；
2. 检查最近备份；
3. 创建发布前备份；
4. 对可能破坏同步的升级暂停写入；
5. 执行数据库迁移；
6. 发布后端；
7. 发布前端和 Service Worker；
8. 健康检查；
9. 执行冒烟测试；
10. 观察错误和同步指标；
11. 解除维护状态。

### 14.3 回滚

- 应用代码可回滚到兼容版本；
- 数据库优先前向修复；
- 必须恢复备份时生成新 sync epoch；
- 客户端强制重新 bootstrap；
- 保存事故和恢复审计。

## 16. 发布门槛

阻止发布：

- P0/P1 缺陷；
- 高危安全问题；
- 数据丢失或静默覆盖；
- 备份不可恢复；
- 数据迁移失败；
- 被撤销设备仍可同步；
- API Key/TOTP/Token 泄漏；
- 断网无法启动或核心编辑丢失；
- AI 能直接修改正式业务状态；
- OpenAPI 与实现严重不一致。

允许带已知问题发布：

- 非核心视觉问题；
- 有明确降级的非主流浏览器问题；
- 不影响数据和安全的轻微性能问题；
- 文档中明确且有后续任务的低风险限制。

## 17. 第一版试运行

正式录入长期数据前进行 14 天试运行：

- 使用复制的 47 天计划；
- 每天至少一次电脑和手机同步；
- 人为断网完成任务和笔记；
- 至少制造并解决一次结构化冲突；
- 上传并恢复附件；
- 配置两个 AI Provider 或一个 Provider + 模拟备用；
- 执行一次手动备份下载；
- 在测试环境恢复一次。

14 天内无数据丢失、无未解释同步异常，才进入长期正式使用。

## 18. 事故级别

```text
P0  数据大规模丢失、账户/密钥泄漏、无法恢复
P1  静默覆盖、认证绕过、备份持续失败、核心离线不可用
P2  单功能失效、有明确恢复方法、无数据损失
P3  轻微错误、体验或文档问题
```

P0/P1 处理后必须：

- 写事故报告；
- 确定时间线和影响；
- 增加回归测试；
- 更新威胁模型或运行手册；
- 验证恢复和监控。
